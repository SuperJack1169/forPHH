Git Merge Introduction (မြန်မာဘာသာပြန်)

Git သည် Distributed Version Control System (DVCS) တစ်ခု ဖြစ်သည်။  
ဥပမာအားဖြင့် ဂျပန်က developer တစ်ဦးနဲ့ New Jersey က developer တစ်ဦးဟာ  
သူသူမမ အလိုအလျှောက်ပြောင်းလဲမှုများကို တစ်ဦးတည်းအဖြစ်မဟုတ်ဘဲ သီးသန့် ပြုလုပ်နိုင်ပြီး  
အချိန်မရွေး သူတို့ ပြောင်းလဲမှုများကို ပေါင်းစည်းနိုင်သည်။  
ဒီအရာအားလုံးကို အလယ်တန်း repository (central server) မလိုဘဲ လုပ်နိုင်သည်။

ဒီအခန်းထဲမှာတော့ အမျိုးမျိုးသော development line များ (branches) ကို  
ဘယ်လိုပေါင်းစည်းရမလဲ ဆိုတာကို သင်ယူသွားမှာ ဖြစ်ပါတယ်။

Merge ဆိုတာ commits history မှာရှိတဲ့ branches များနှစ်ခု (သို့မဟုတ်) များများကို  
တစ်ခုတည်းအဖြစ် ပေါင်းစည်းခြင်း ဖြစ်သည်။  
အများဆုံးအခါတွေမှာတော့ branch နှစ်ခုကိုပဲ merge လုပ်ကြသော်လည်း —  
Git က သုံးခု၊ လေးခု သို့မဟုတ် များများကိုတစ်ပြိုင်နက်တည်း merge လုပ်နိုင်သည်။

Git မှာ merge လုပ်တဲ့အခါ —  
အဲဒီ branch များအားလုံးကို တစ်ခုတည်းသော repository အတွင်းမှာရှိရမယ်။  
ဒါဟာ "Git မှာ merge လုပ်ဖို့ဆိုရင်၊ ပေါင်းစည်းမယ့် branch များအားလုံးဟာ တူညီတဲ့ repo ထဲမှာရှိဖို့လိုတယ်" ဆိုတာကို မြင်ပေးရမယ်။  
အဲ့ဒီ branch များကို အဲ့ဒီ repository ထဲကို ဘယ်လိုယူလာတာလဲဆိုတာက အရေးမကြီးပါ။  
(Chapter 12 မှာ ကြည့်ရမယ့်အတိုင်း Git မှာ remote repository များကို reference လုပ်ဖို့နဲ့  
remote branch များကို ယူသွင်းဖို့ mechanism များရှိပါတယ်။)

တစ်ခုသော branch မှ ပြောင်းလဲမှုများသည် တခြား branch မှ ပြောင်းလဲမှုများနှင့် မပဋိပက္ခပါက —  
Git က merge result ကို ကိုယ်တိုင်တွက်ချက်ပြီး  
ပြောင်းလဲမှုပေါင်းစည်းထားသော commit အသစ်တစ်ခုကို ဖန်တီးပေးသည်။

ဒါပေမယ့် တူညီတဲ့ file ထဲမှာ တူညီတဲ့ line ကို နှစ်ဖက်စလုံးက ပြောင်းချင်တဲ့အခါ conflict ဖြစ်တတ်သည်။  
ဒီလို conflict ဖြစ်တဲ့အခါ Git က မိမိတင်ပြချက်ဖြင့်အဖြေမပေးပါ။  
Git ဟာ ပြဿနာဖြစ်နေတဲ့ change များကို “unmerged” ဟု index ထဲတွင်မှတ်သားပြီး  
ဖြေရှင်းမှုကို developer ကိုယ်တိုင် (မင်း) ဆောင်ရွက်ဖို့ထားခဲ့သည်။  
Git က မိမိအလိုအလျှောက် merge မလုပ်နိုင်တဲ့အခါမှာလည်း  
ဖြေရှင်းပြီးတဲ့နောက် commit အသစ်ကို မင်းကိုယ်တိုင်လုပ်ဖို့လိုတယ်။
